# Ajax 是否遵循同源策略

是，同源策略的本质是一种约定，可以说 web 的行为就是构建在这种约定之上的。就好比我们人类的行为必须受到法律的约束一样。同源策略的目的就是限制不同源的 document 或者脚本之间的相互访问，以免造成干扰和混乱。ajax 太灵活了，各种请求说法就发，如果没有同源策略的限制，发到哪里都行，只要你构造好参数和请求路径，那人人都是黑客了，这样会导致各种敏感数据的泄露。

# 怎么确定一个接口地址有没有 CSRF 漏洞

如果我是一个第三方网站，我去访问一个接口地址，它也能响应 200 的话，则可能存在 CSRF

# 怎么防御 CSRF

1. 禁止第三方的网站对我发起请求，通过 referer 来校验

2. 第一种方法还是有漏洞，因为，http header 的字段可以任意修改

3. 因为 CSRF 是获取不到 Cookie 的，所以可以把 Cookie 加密，然后发给服务器，服务器再用相同的加密算法加密一次，然后对比，如果一样，就证明是用户本人发起的

4. 服务器端生成 token

5. 执行敏感操作的时候二次验证

6. 加验证码

# SSRF

服务端请求伪造

原理：由于某些应用需要调用其他服务器的资源，需要传入一个目标地址，通过控制这个目标地址，就可以造成一些危害

危害：

1. 可以服务器进行端口扫描

2. 向内部任意主机的任意端口发送精心构造的 payload，攻击运行在内网的应用程序

3. 攻击内网的 web 应用，主要是使用 GET 参数就可以实现的攻击

4. 利用 file 协议读取本地文件等

如果后端使用的是 curl 函数，那么就可能执行 ftp,ftps,http,https,telnet,dict,file,ldap 等协议

SSRF 漏洞的寻找

1）分享：通过 URL 地址分享网页内容

2）转码服务

3）在线翻译

4）图片加载与下载：通过 URL 地址加载或下载图片

5）图片、文章收藏功能

6）未公开的 api 实现以及其他调用 URL 的功能

7）从 URL 关键字中寻找

SSRF 验证

1. 抓包分析请求是否是由服务器发起的

2. 在页面源码寻找访问的资源地址，所有有外部资源调用的地方都有可能存在 SSRF 漏洞

SSRF 防御

1. 禁用不需要的协议

2. 同一错误信息处理，防止端口探测

3. 禁止 302 跳转，not found

4. 设置 url 白名单或限制内网 ip
